# Plan: Adamas Portal Auth Integration

## Goal
Replace the local-database login with real authentication against the Adamas student portal (`https://adamasknowledgecity.ac.in/student/login`). After login, fetch student info (name, department, school, year) from the portal dashboard.

## Portal Login Analysis

The Adamas portal is a standard Laravel app:

- **Login URL:** `POST https://adamasknowledgecity.ac.in/student/login`
- **Form fields:** `registration_no`, `password`, `_token` (CSRF), `login` (submit)
- **CSRF:** Hidden `_token` field — must GET the page first to extract it
- **Session:** Cookie-based (Laravel session) — need to persist cookies across requests

## Changes Required

### 1. New Erlang FFI module: `services_adamas_ffi.erl`
**File:** `apps/synapse/src/services_adamas_ffi.erl`

Three FFI functions:
- `fetch_csrf_token() -> {ok, Token} | {error, Reason}` — GETs the login page, parses the `_token` value from HTML
- `portal_login(Token, RegistrationNo, Password) -> {ok, Cookie} | {error, Reason}` — POSTs credentials with CSRF token, returns session cookie on success
- `fetch_student_info(Cookie) -> {ok, Json} | {error, Reason}` — GETs the dashboard/student profile page with the session cookie, extracts student info

Implementation uses OTP `httpc` (already used by `services_llm_ffi.erl`). Must handle:
- SSL (the portal uses HTTPS)
- Cookie persistence across requests
- HTML parsing to extract `_token` and student data

### 2. Update `identity.gleam`
**File:** `apps/synapse/src/services/identity.gleam`

- Add FFI bindings for the three new functions
- Add `portal_login(registration_no, password) -> LoginResult` that:
  1. Fetches CSRF token
  2. Posts credentials to the portal
  3. On success, fetches student info from dashboard
  4. Creates/upserts the student in local DB (for session management)
  5. Creates a Redis session
  6. Returns `Authed(session_id, school, department, year)`
- Fix `parse_login_body` to accept `registration_no` field (matching the portal's field name)

### 3. Update `synapse.gleam` login handler
**File:** `apps/synapse/src/synapse.gleam`

- Change `/login` handler to call `identity.portal_login()` instead of the local-only flow
- The existing session creation + cookie flow stays the same

### 4. Fix frontend field name mismatch
**File:** `apps/mobile/App.tsx`

- Change `registration_number` to `registration_no` in the login POST body (matching what the portal uses)

### 5. Wire up session storage on frontend
**File:** `apps/mobile/App.tsx`

- On successful login, store the session cookie (using `document.cookie`)
- Navigate to a chat/dashboard view (or at minimum show success state)
- Read the session cookie on subsequent requests

## Risks
- **Portal HTML changes:** The scraping approach is fragile — if Adamas changes their login page HTML, the CSRF extraction or student info parsing will break. Mitigation: keep parsing logic isolated in one FFI module for easy updates.
- **Rate limiting:** The portal may rate-limit or block repeated login attempts. Mitigation: add reasonable retry limits.
- **No official API:** This is scraping, not an official integration. The portal could block automated requests. Mitigation: use reasonable User-Agent and request timing.
